## Repo Roles

| Path | Role | Modification Authority |
|------|------|----------------------|
| `pages/` | Compiled mirror output — source for Eleventy build | Only written by `build-mirror-from-snapshot.sh` |
| `WEB/base44-snapshot/<ts>/` | Immutable historical snapshot | Never modified after creation |
| `WEB/base44-snapshot/latest/` | Promoted working input (symlink) | Updated only by `promote-snapshot.sh` |
| `WEB/scripts/` | Pipeline scripts | Modified only by operational contract update |
| `WEB/contracts/` | Operational contracts | Modified only with documented reason |
| `WEB/logs/` | Run logs and manifests | Append-only after run completion |
| `docs/mirror-validation/` | Validation artifacts | Written by `build-mirror-from-snapshot.sh` |
| `_site/` | Build output | Generated by Eleventy — never committed |
| `_includes/` | Layout templates | Modified only for structural changes |
| `_data/` | Site data | Modified only with documented reason |

## Execution Order

Every release must follow this sequence. No step may be skipped.

| Step | Stage | Script / Tool | Required Output |
|------|-------|--------------|----------------|
| 1 | Push | MCP: `edit_base44_app` | Push manifest in `WEB/logs/push-runs/` |
| 2 | Extract | `extract-base44-pages.js` | Timestamped snapshot in `WEB/base44-snapshot/<ts>/` |
| 3 | Promote | `promote-snapshot.sh` | `latest` symlink + `promotion_manifest.md` |
| 4 | Compile | `build-mirror-from-snapshot.sh` | Compiled pages in `pages/` |
| 5 | Internal validate | (within build script) | `hard_validator_report.md` + `compile_manifest.md` |
| 6 | External validate | `validate-search-console.sh` + `check-url-indexation.sh` + `check-sitemap.sh` | `external_validation_report.md` |
| 7 | Eleventy build | `npx @11ty/eleventy` | `_site/` |
| 8 | Deploy | Host-specific deployment | Routes live on krayu.be |
| 9 | Cache purge | Cloudflare API or Dashboard | Cache cleared for updated routes |
| 10 | Post-release check | Manual or automated | URLs accessible, SSL valid, sitemap updated |

If any step fails: stop. Do not proceed to the next step without resolving the failure.

## Release Checkpoints

| Checkpoint | Pass Condition | Failure Action |
|------------|---------------|----------------|
| Push complete | `status = ready`, manifest written | Stop. Do not extract. |
| Extraction complete | 3+ PASS, 0 FAIL in capture_manifest | Stop. Do not promote. |
| Promotion valid | All 7 promotion validation checks PASS | Stop. Do not compile. |
| Compile valid (strict) | All 10 compile checks PASS | Stop. Do not build. |
| Hard validators | All 8 semantic validators PASS or PARTIAL | Evaluate. FAIL blocks build. |
| External validation | PASS or NOT_CONFIGURED | NOT_CONFIGURED: proceed with warning. FAIL: investigate. |
| Eleventy build | Exit code 0 | Stop. Do not deploy. |
| Deploy | All routes accessible | Stop. Cache purge not needed. |

## Rollback Rules

**Rollback trigger conditions:**
- A deployed page is returning 404, 500, or incorrect content
- A canonical core page has been unintentionally modified
- A baseline protection violation is discovered post-deploy
- The Eleventy build deployed malformed output

**Rollback procedure:**
1. Identify the prior working `pages/` state from git history
2. `git checkout <prior-commit> -- pages/`
3. Run `npx @11ty/eleventy` to rebuild from prior state
4. Redeploy `_site/`
5. Purge cache
6. Document the rollback in `WEB/logs/push-runs/` as a rollback manifest

**Rollback does NOT:**
- Alter historical snapshots in `WEB/base44-snapshot/<ts>/`
- Modify the `latest` symlink (unless the promoted snapshot was the source of the problem)
- Rewrite push manifests

## Artifact Retention Rules

| Artifact | Retention | Notes |
|----------|-----------|-------|
| Timestamped snapshots | Permanent — immutable | Never delete |
| Push manifests | Permanent — immutable | Never delete |
| `latest` symlink | Mutable — updated per promotion | Points to current working input |
| `pages/*.md` | Mutable — updated per compile | Source for Eleventy build |
| `docs/mirror-validation/` | Mutable — updated per compile | Append-only for new sections |
| `_site/` | Ephemeral — regenerated per build | Not committed to repo |
| Git history | Permanent | All pages/ changes tracked via git |

## Release Documentation Requirement

Every release must be traceable through:

1. Push manifest (WEB/logs/push-runs/)
2. Capture manifest (WEB/base44-snapshot/<ts>/capture_manifest.md)
3. Promotion manifest (WEB/base44-snapshot/<ts>/promotion_manifest.md)
4. Compile manifest (docs/mirror-validation/compile_manifest.md)
5. Git commit covering the pages/ changes

A release without a complete manifest chain is an ungoverned release and must be treated as incomplete.
